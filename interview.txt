Walk me through an end-to-end RAG pipeline, from user query to final answer
When a user asks a question, the first thing that happens is pretty basic hygiene. I clean up the input, log it, and do a quick check on what kind of question it is. Is this a factual lookup, a “how do I” question, or something more open ended? That decision matters later for how strict I want the model to be and how much creativity I allow.

Next, I usually rewrite the query. Real users don’t ask clean search questions. They ask things like “does this still work?” or “what’s the policy here?” So I’ll often use a small LLM step to turn that into a clearer, more explicit search query. For example, it might add the product name, expand abbreviations, or make the question more specific. This step alone can make retrieval much better.

Then comes retrieval, which is really the heart of RAG. I already have a corpus of documents that I’ve preprocessed. That means I’ve split them into chunks that are small enough for the model, added some overlap so context is not lost, and stored metadata like source and date. Each chunk has an embedding, and usually I also index the raw text for keyword search.

At query time, I typically run two searches in parallel. One is a keyword search like BM25, which is great when the user uses exact terms. The other is a dense vector search using embeddings, which is better for semantic matches and paraphrased questions. I merge those results and take the top candidates.

That first retrieval step usually gives me too much. So I re-rank. I take maybe the top 50 chunks and pass them through a stronger model, often a cross-encoder, that looks at the query and chunk together and scores relevance more accurately. From there I keep the top few that are both relevant and not redundant.

Now I build the context that will go into the LLM. I select the best chunks that fit within the context window and attach their source information. This part is important. If I stuff in too much text, the model gets confused. If I put in too little, it hallucinates. There’s some trial and error here.

Then I construct the prompt. I’m very explicit. I tell the model that these are the only sources it should use, and that if the answer is not supported by them, it should say so. I also tell it how to answer, like concise, neutral tone, and with citations. For factual questions I usually keep the temperature low.

The LLM then generates the answer using that grounded context. After that, I often run a lightweight check. I look for claims that don’t seem to map back to any retrieved chunk, or missing citations. If something looks off, I either ask the model to be more conservative or return an “I couldn’t find a reliable answer” response.

Finally, I format the output for the user. That might mean a short direct answer at the top, followed by details and links to the original sources. I also log everything: what was retrieved, what was shown to the model, latency, and user feedback if available.

In production, the work doesn’t stop there. I monitor retrieval quality, answer correctness, and drift in user queries. I refresh documents, re-embed when content changes, and tune chunking and prompts over time. RAG looks simple on paper, but most of the real work is in these small decisions and trade-offs.
